# Техническое задание — MeetSync

## Название проекта

MeetSync

---

## Назначение проекта

Распределённый веб-сервис для организации совместного досуга групп друзей.

Сервис позволяет:

- создавать компании (группы);
- вести банк идей совместного отдыха;
- отмечать занятость каждого участника в календаре;
- автоматически подбирать подходящие даты;
- проводить голосования за идеи;
- создавать встречи с RSVP (пойду/не пойду/возможно);
- отправлять уведомления через Telegram;
- рекомендовать события на основе открытых API.

Система состоит из независимых микросервисов (12 сервисов + инфраструктура).

## Стек технологий

### Бэкенд

- Python 3.13
- FastAPI
- SQLAlchemy 2 (async)
- Alembic
- httpx (межсервисные запросы)
- Pydantic (валидация схем)

### Фронтенд

- React 19
- TypeScript
- Vite
- Material UI 6
- nginx (раздача SPA + прокси API)

### База данных

PostgreSQL 16

Обоснование: сложные связи, множественные JOIN, транзакции, данные календарей и голосований — реляционная модель оптимальна.

### Кэш

Redis 7

Используется для:

- кэширования
- хранения JWT blacklist (при выходе)
- rate limiting (60 запросов/мин)

### Очередь сообщений

RabbitMQ 4

Сервисы взаимодействуют через события. Используется topic exchange `meetsync.events` — надёжная доставка, dead-letter очереди, персистентные обменники, легко добавлять подписчиков.

## Архитектура

```
Frontend (:3000) → Gateway (:8000) → Auth / User / Group / Calendar / Ideas / Meeting / Voting / Recommend / Notification / Telegram / Scheduler
                                                                                    ↓
                                                                        PostgreSQL + Redis + RabbitMQ
                                                                        Prometheus + Grafana
```

Все сервисы независимы:

- У каждого своя директория `app/`, `Dockerfile`, `requirements.txt`
- Каждый сервис владеет своими таблицами в БД
- Конфигурация из единого `.env` файла
- Общая библиотека `shared/` — установлена как editable pip-пакет во всех сервисах

### Коммуникация

- **Синхронная (REST):** Фронтенд → nginx (TLS termination, HTTPS) → Gateway → Микросервис. Все HTTP-запросы проходят через Gateway. nginx терминирует SSL и проксирует на Gateway по HTTP.
- **Асинхронная (RabbitMQ):** Микросервис → Exchange → Queue → Микросервис. События публикуются для побочных эффектов.
- **WebSocket:** Notification Service поддерживает постоянные соединения для real-time уведомлений.
- **Telegram Bot:** HTTPS к Telegram API (polling-режим для локальной разработки, webhook для продакшена).

## API Gateway

Единая точка входа. Все запросы идут через него.

- FastAPI + httpx.AsyncClient (timeout 30s)
- Роутинг: `/api/v1/{service}/*` → `http://{service}:{port}/*`
- JWT passthrough, rate limiting (60 req/min), CORS
- Swagger: http://localhost:8000/docs

## Сервисы

### Auth Service (:8001)

**Эндпоинты:** `/register`, `/login`, `/refresh`, `/telegram`, `/logout`, `/me`

- bcrypt для хеширования паролей
- JWT access token (30 мин) + refresh token (7 дней)
- Хранение чёрного списка JWT в Redis (при logout)
- Привязка Telegram аккаунта через deep link

**Публикует:** `UserRegistered`

### User Service (:8002)

**Эндпоинты:** `GET /me`, `PATCH /me`

- Имя, email, Telegram ID, username, аватар
- Часовой пояс
- Настройки

### Group Service (:8003)

**Эндпоинты:** CRUD групп, `/join`, `/{id}/members`, `/{id}/refresh-invite`

- Роли: OWNER (создатель), ADMIN, MEMBER
- Инвайт-код (уникальный, алфавитно-цифровой)
- Пользователь может состоять в нескольких группах

**Публикует:** `UserJoinedGroup`, `UserLeftGroup`

### Calendar Service (:8004)

**Основная сущность проекта.**

- Статусы: `free`, `busy`, `maybe`
- Эндпоинты: `POST /availability`, `GET /availability/me`, `GET /availability/group`, `GET /availability/user/{id}`, `DELETE /availability/{id}`

**Публикует:** `AvailabilityUpdated`, `AvailabilityConfirmed`

### Ideas Service (:8005)

Банк идей (backlog). Каждая идея содержит:

- Название, описание, категория, стоимость, фото, теги, локация
- Кто предложил, дата создания
- Архивация

**Реакции (4 эмодзи):** 👍❤️🔥👎

**Комментарии:** текст + автор + дата

**Публикует:** `IdeaCreated`, `IdeaArchived`

### Meeting Service (:8011)

**Эндпоинты:** CRUD встреч, `/{id}/rsvp`

- Создание встречи: group_id, заголовок, описание, дата, время, локация
- Опциональная привязка к идее (`idea_id`)
- RSVP-статусы: `going`, `not_going`, `maybe` (по умолчанию `maybe`)
- Удалить встречу может только создатель
- Просмотр списка участников с их RSVP-статусами

### Voting Service (:8006)

**Эндпоинты:** CRUD голосований, `/{id}/vote`

- Типы: `random` (случайные), `popular` (самые популярные), `category` (по категории)
- Авто-завершение по таймеру, победитель определяется по голосам

**Публикует:** `VotingStarted`, `VotingFinished`

### Notification Service (:8008)

**Эндпоинты:** `GET /notifications`, `POST /{id}/read`, `POST /read-all`

- WebSocket для real-time доставки
- Хранение истории уведомлений в БД

**Потребляет:** все основные события

### Telegram Service (:8009)

- **Режимы:**
  - Long-polling (локальная разработка) — `getUpdates` с таймаутом 30с, httpx timeout 35с
  - Webhook (продакшен) — требуется публичный HTTPS URL
- **Команды:** `/start` (приветствие), `/start <code>` (вход в группу)
- Inline-кнопки, динамическая клавиатура

**Потребляет:** `VotingFinished`, `MeetingPossible`, `ReminderNeeded`, `NotificationSent`

### Recommendation Service (:8007)

**Эндпоинты:** `GET /recommendations`, `GET /weather`

- Открытые API: Ticketmaster, OpenWeather, Google Places (настраиваются через API-ключи)
- Фильтрация: по городу, категории, цене, дате

### Scheduler Service (:8010)

Фоновый планировщик:

- Ежедневная проверка доступности
- Предложение встречи, когда N+ участников свободны
- Напоминания отметить доступность

**Потребляет:** `AvailabilityUpdated`, `AvailabilityConfirmed`

**Публикует:** `MeetingPossible`, `ReminderNeeded`, `MeetingCancelled`

## Ключевые сценарии

### 1. Идея → Встреча → RSVP

```
Пользователь A: создаёт идею
Пользователь B: ставит реакции
Пользователь A/B: нажимает "Запланировать встречу" → Meeting Service создаёт встречу
Участники: отмечают RSVP (going / not_going / maybe)
```

### 2. Календарь доступности

```
Пользователь: отмечает день как Free/Busy/Maybe
              → Calendar Service сохраняет
              → Участники группы видят общий календарь
              → Scheduler предлагает оптимальные даты
```

### 3. Голосование

```
Админ: создаёт голосование (random / popular / category)
Участники: голосуют в течение таймера
Победитель: определяется автоматически
Победитель: может быть запланирован как встреча
```

## Фронтенд

### Технологии

- React 19 + TypeScript + Vite
- Material UI 6 — кастомная тема с дизайн-системой
- Inter — основной шрифт

### Дизайн-система

- **Цвета:** нейтральная палитра (50–950), акцент — тёплый индиго (#6366F1 → #818CF8)
- **Тёмная/светлая тема:** переключение с сохранением в localStorage, автовыбор `prefers-color-scheme`
- **Компоненты:** единый стиль для Button, Card, Dialog, Chip, Tabs, TextField, AppBar, Drawer и др.
- **Layout:** коллапсируемый сайдбар (260px ↔ 72px), стеклянная AppBar (backdrop-filter blur)
- **Адаптивность:** мобильная версия с bottom sheet для деталей календаря, компактные ячейки 48px

### Страницы

- **Dashboard** — следующий голос, следующая встреча, последние идеи, статистика
- **My groups** — список групп, создание, вступление по коду
- **Group detail** — участники, календарь группы, идеи, встречи
- **Calendar** — большой календарь: зелёный (свободен), красный (занят), жёлтый (возможно), серый (не отмечено)
- **Ideas** — банк идей с реакциями и комментариями
- **Voting** — активные и завершённые голосования
- **Meetings** — список встреч с RSVP
- **Notifications** — WebSocket + история
- **Recommendations** — внешние события и погода
- **Settings** — профиль, настройки, тема
